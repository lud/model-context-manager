# Subcontexts feature

Added subcontexts support to MCM, allowing files across related doctypes to be
grouped under feature-specific directories. Each subcontext gets its own
numbering namespace while non-scoped doctypes stay at the project level.

## What changed

- **Schema** (`src/lib/project.ts`): Added `SubcontextsSchema` with `dir` and
  `doctypes` fields. Added `ResolvedDoctypeEntry` (extends `DoctypeEntry` with
  `inSubcontext`). Updated `ResolvedProject` with `currentSubcontext` field.
  `loadProjectOrFail` now accepts an optional `subcontext` parameter to resolve
  managed doctype dirs through the subcontext directory.

- **Global config** (`src/lib/global-config.ts`): Added `currentSubcontexts`
  map (projectDir → subcontext name) with `getCurrentSubcontext` and
  `setCurrentSubcontext` helpers.

- **Subcontext helpers** (new `src/lib/subcontext.ts`): `listSubcontexts`,
  `resolveSubcontextArg` (number or fuzzy match), `nextSubcontextDirName`
  (000-padded numbering).

- **`mcm sub` command** (new `src/commands/sub.ts`): Actions: `add` (create
  subcontext + managed doctype subdirs), `switch` (by number or fuzzy name),
  `list` (with `*` marker for current), `current`/`which` (print path).

- **`--sub` flag** on `list`, `new`, `next` commands: Passes through to
  `getProject({ sub })`. Commands abort with a clear message when a managed
  doctype is used without an active subcontext.

- **JSON schema** updated with `subcontexts` property.

- **Tests**: 38 new tests covering subcontext helpers, project resolution,
  sub command actions, and subcontext guards on existing commands.
