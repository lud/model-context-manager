# Global Per-Doctype Sequence Numbers Across Subcontexts

When a doctype is managed by subcontexts, each subcontext previously had its own independent file sequence starting at 001. This caused ambiguity when referencing files by number across subcontexts.

## Changes

### `src/lib/project.ts`
Added `DoctypeFileEntry` type and `listDoctypeFilesAcrossSubcontexts` function. For non-subcontext doctypes it reads the single directory; for subcontext-managed doctypes it scans all subcontext dirs and returns entries for each, with empty file lists for missing dirs.

### `src/lib/sequence.ts`
Added `DirRename` type and `computeGlobalRenames` function. Collects files from all `DoctypeFileEntry` entries, sorts globally by `(seq, slug)`, assigns consecutive numbers, and returns only the changed entries with their directory.

### `src/commands/next.ts` and `src/commands/new.ts`
Replaced the local `readdirSync(entry.dir)` call with `listDoctypeFilesAcrossSubcontexts` + `flatMap`. The global max sequence is now used to compute the next filename. Files are still written to the active subcontext's directory.

### `src/commands/seqfix.ts`
Added an `inSubcontext` branch that uses `listDoctypeFilesAcrossSubcontexts` + `computeGlobalRenames`. Displays paths relative to the subcontexts dir (e.g. `001.feature-a/notes/003.design.md → 004.design.md`) so the user sees which subcontext each file belongs to. Two-phase rename is applied per-directory.

### `src/lib/project.test-helpers.ts`
Updated `mockProject` to also spy on `listDoctypeFilesAcrossSubcontexts` with a real filesystem-reading implementation, since the auto-mock replaces it with `vi.fn()` returning undefined.

### Test fixture: `test/fixtures/doctypes/multi-subcontext/`
Two subcontexts with intentionally colliding/gapped sequences, plus an empty third subcontext (no notes dir) for missing-dir coverage.
