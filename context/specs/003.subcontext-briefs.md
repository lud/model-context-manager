# Subcontext Briefs

## Problem

MCM supports subcontexts (e.g. features) as directories containing managed doctypes
(e.g. tasks, specs). But the current model has workflow gaps:

- You can list the next open task, but you don't know which spec it relates to.
- You can close a task, but there's no way to know if all tasks for a spec are done.
- Completing all specs in a feature doesn't surface "what's the next feature?"
- Subcontexts have no metadata (status, tags, properties) — they're just directories.

Meanwhile, the generic doctype system is valuable for custom setups (ADRs, notes,
guides, etc.) and should not be complicated by workflow concerns.

## Design Goals

1. Give subcontexts a file identity with frontmatter (status, tags, properties).
2. Unify subcontext management with existing doctype commands where possible.
3. Keep the generic doctype system simple and unopinionated.

## Solution: Briefs

A **brief** is a markdown file that embodies a subcontext. It lives inside the
subcontext directory and is named after it:

```
features/
  001.add-auth/
    001.add-auth.md        <-- brief
    tasks/
      001.design-api.md
      002.write-tests.md
  002.payments/
    002.payments.md        <-- brief
    tasks/
      001.stripe-setup.md
```

The brief has frontmatter like any other managed file (status, tags, properties),
which enables filtering and workflow tracking at the subcontext level.

### Why nested (inside the directory), not sibling (next to it)?

We considered placing the brief as a sibling file next to the subcontext directory
(`features/001.add-auth.md` alongside `features/001.add-auth/`). This was rejected
for several reasons:

- **Rename safety:** With the nested approach, renaming the directory moves the brief
  with it. Users can't accidentally orphan the brief by renaming only the directory
  or only the file. If the file inside is renamed but not the directory (or vice
  versa), the mismatch is detectable by comparing slugs.
- **Colocation:** The brief lives alongside the tasks it describes, not separated by
  file explorer sorting (which typically groups directories above files).
- **Simpler mental model:** The directory *is* the subcontext; the brief *describes*
  it. One is inside the other.

### Why not `index.md`?

Every subcontext would have an `index.md`, making IDE fuzzy search useless. The
sequenced, slugged name (e.g. `001.add-auth.md`) is unique across the project.

## Subcontext as a Doctype

The subcontext doctype is configured like any other doctype but is referenced by a
top-level config key. This replaces the current `subcontexts` config shape.

### Current config

```json
{
  "doctypes": {
    "specs": { "dir": "specs" },
    "tasks": { "dir": "tasks" }
  },
  "subcontexts": {
    "dir": "features",
    "doctypes": ["specs", "tasks"]
  }
}
```

### New config

```json
{
  "doctypes": {
    "features": { "dir": "features" },
    "tasks": { "dir": "tasks" }
  },
  "subcontextDoctype": "features",
  "managedDoctypes": ["tasks"]
}
```

- `subcontextDoctype` points to a doctype defined in `doctypes`.
- `managedDoctypes` lists doctypes whose files live inside subcontext directories.
- The subcontext doctype itself must not appear in `managedDoctypes`.
- `"sub"` is a reserved alias: `mcm list sub` resolves to the configured subcontext
  doctype. `"sub"` is not allowed as a doctype name.

## Command Unification

Most `sub` subcommands become regular doctype commands:

| Before                    | After                              |
| ------------------------- | ---------------------------------- |
| `mcm sub add "add auth"`  | `mcm new features "add auth"`      |
| `mcm sub list`            | `mcm list features` or `mcm list sub` |
| `mcm sub seqfix`          | `mcm seqfix features`              |
| (not possible)            | `mcm list features --open --first` |
| (not possible)            | `mcm close <brief-path>`           |
| `mcm sub switch`          | `mcm sub switch` (kept)            |
| `mcm sub current`         | `mcm sub current` (kept)           |

The `sub` command shrinks to `switch` and `current` only.

## Impact on Existing Commands

### `list`

The subcontext doctype uses a different scan pattern: instead of reading files
directly from `entry.dir`, it reads subdirectories and finds the eponymous `.md` file
inside each one.

This logic should be centralized in a function that `list`, `new`, `next`, and
`seqfix` all use, so the branching doesn't leak into every command.

### `new`

For the subcontext doctype, `mcm new features "add auth"` must:

1. Compute the next sequence number (from existing subcontext directory names).
2. Create the directory (`features/003.add-auth/`).
3. Create the brief file inside (`features/003.add-auth/003.add-auth.md`).
4. Create subdirectories for each managed doctype (`tasks/`, etc.).
5. Set the new subcontext as current.

### `seqfix`

When renumbering the subcontext doctype, both the directory and the brief file inside
must be renamed in lockstep. The directory is renamed first (two-phase as today), then
the brief file inside is renamed to match.

### `close`

No changes needed. Users pass the brief file path explicitly.

### `next`

Needs the same scan pattern change as `list` — read subcontext directories to find
existing briefs for sequence calculation.

### `sub switch` / `sub current`

No changes to behavior. These remain the only `sub` subcommands.

### `sub seqfix` / `sub add` / `sub list`

Removed. Replaced by the unified doctype commands.

## Mismatch Detection

Because the brief is nested inside its subcontext directory, we can detect when a user
has manually renamed one but not the other:

- On any subcontext scan, compare the directory slug to the brief file slug.
- If they differ, emit a warning (e.g. "brief 001.add-auth.md does not match
  directory 002.new-name — consider renaming").

## Init Command

The `init` command and its templates need updating:

- The "developer" template should use the new config shape (`subcontextDoctype`,
  `managedDoctypes`) instead of the old `subcontexts` block.
- Custom setup flow should offer the option to designate a doctype as the subcontext
  doctype.

## Open Questions

- Should `mcm new features "foo"` automatically switch to the new subcontext? The
  current `sub add` does this. Probably yes, to preserve the workflow.
- Should listing managed doctypes without an active subcontext show a hint about
  switching, or silently list nothing? Current behavior aborts with an error for `new`
  but falls back to global scan for `list`.
